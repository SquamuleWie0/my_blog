# 从零搭建 Learning Agent

>FastAPI、LLM 接入、Markdown 落盘与 Obsidian 集成全流程实践
## 摘要

这篇文章完整记录了我从零开始搭建一个 Learning Agent 的全过程。这个项目的目标不是单纯做一个“能聊天”的接口，而是做一个能够真正服务于我自己的学习与写作流程的小工具：接收我的项目记录、学习草稿或开发过程，调用大语言模型进行整理，生成结构化的 Markdown 技术文档，并自动保存到我的 Obsidian 笔记目录中。

整个项目经历了多个阶段：从 Git 仓库初始化、FastAPI 服务搭建、接口联调，到 LLM 接入、Markdown 文件写入、Obsidian 路径配置、文件命名优化，以及后续的 Prompt 约束和简单前端页面构建。过程中也踩了不少坑，比如 pip 命令不可用、端口占用、`.env` 配置不匹配、内容生成但未写入文件、文件名固定和双日期叠加等问题。

这篇文章会按照阶段拆解整个实现过程，并补充每一步背后的原因、遇到的问题、当前成果和后续优化方向。

---

# 一、项目背景：我为什么要做这个 Agent

我做这个项目，不是为了“做一个 AI Demo”，而是为了构建一个真正能融入自己学习流程的工具。

我在平时学习和做项目时，经常会留下很多零散内容，比如：

- 当天的开发操作记录
- 遇到的报错和解决过程
- 对某个技术点的理解
- 对下一步的想法
- 想整理成博客但还很混乱的草稿

这些内容本身是有价值的，但如果靠手动再去整理成文章，会非常耗时。于是我希望有一个系统，能够完成这样一条链路：

```text
我输入零散记录 -> Agent 帮我整理 -> 输出 Markdown 技术文档 -> 自动保存到我的笔记目录
```

这个需求本质上不是“聊天”，而是一个小型内容生产流程。所以我给这个项目定下的目标就是：

> 做一个能够把我的学习记录和开发过程自动转成技术博客草稿的 Agent。

---

# 二、先理解 AI Agent：它到底是什么

在真正开始做项目之前，我先建立了一个比较清晰的认知：

> AI Agent 不是“接一个大模型 API”这么简单，而是一个由多个环节共同组成的小系统。

我把它理解成：

```text
AI Agent = 大模型 + Prompt + 工具 + 流程 + 状态
```

下面分别解释这几个组成部分。

## 1. 模型：Agent 的“大脑”

模型一般就是大语言模型，比如 OpenAI、DeepSeek、Moonshot 等。

它负责做这些事：

- 理解我的输入
- 总结内容
- 重组结构
- 生成标题
- 改写成技术博客风格
- 提炼问题与解决过程

如果没有模型，Agent 只是一个普通后端接口，不具备“智能整理”的能力。

---

## 2. Prompt：Agent 的“行为规则”

Prompt 决定模型应该怎么写、写成什么风格、输出成什么结构。

比如我后面在项目里就通过 Prompt 约束了这些规则：

- 第一行必须是一级标题
- 标题不要带日期
- 不要使用 emoji 和图标
- 内容要偏工程实践记录
- 尽量保留真实操作过程和问题分析
- 输出适合直接保存成 Markdown 博客

如果模型是大脑，那 Prompt 就是“大脑的工作说明书”。

---

## 3. 工具：让 Agent 不只是“会说”，还能“会做”

模型本身只能生成文本，但 Agent 想要真正可用，就要具备工具能力。

这个项目里，我最早接入的工具就是：

- 读取配置
- 写入 Markdown 文件
- 保存到 Obsidian 指定路径

也就是说，我的 Agent 不是只在接口里返回一段话，而是能真正把内容“落盘”。

---

## 4. 流程：让任务可靠执行

一个真正可用的 Agent，不能只是“用户输入，模型胡乱输出”，而需要有清晰的执行步骤。

我当前这个项目的主流程是：

```text
用户输入 -> FastAPI 接口 -> LLM 生成内容 -> Markdown 组装 -> 文件写入 -> 返回路径
```

有了清晰流程后，项目就不再是单点能力，而是一个完整链路。

---

## 5. 状态：保存中间信息和上下文

状态可以理解成这个 Agent 在执行过程中需要记住的东西，比如：

- 当前用户输入
- 当前保存目录
- 当前模型配置
- 当前生成结果
- 文件路径
- 文件名    

虽然我现在做的还是第一版，状态管理还比较轻，但本质上它已经存在了。

# 三、阶段一：项目初始化与 Git 管理

## 这一阶段做了什么

项目一开始，我先做的并不是模型接入，而是先把整个项目仓库初始化好。

我执行了：

```bash
git init
```

然后查看状态：

```bash
git status
```

这一步的意义是让整个项目进入可追踪状态，后面每一步改动都可以被版本管理记录下来。

之后我完成了第一次提交：

```bash
git add .
git commit -m "init project"
```

再把仓库推到 GitHub。

---

## 为什么这一步重要

很多人做项目时会直接开始写代码，但我现在越来越意识到：

> 工程项目从第一天起就应该进入版本管理。

如果没有 Git，整个过程会很混乱，也不利于后面写技术总结。

---

## 遇到的问题

我最开始往远程推送时，直接用了示例地址：

```bash
git remote add origin https://github.com/yourname/blog-agent.git
```

结果报错：

```bash
remote: Repository not found.
fatal: repository not found
```

问题原因很简单：这个地址只是示例，不是我的真实仓库地址。

### 解决方式

我重新创建了自己的 GitHub 仓库，然后使用真实地址：

```bash
git remote remove origin
git remote add origin https://github.com/我的用户名/learning-agent.git
git push -u origin main
```

---

## 这一阶段的成果

到这里，项目已经有了：

- 本地 Git 仓库
- 远程 GitHub 仓库
- 可追踪的版本历史

这为后面的迭代打下了基础。

# 四、阶段二：搭建 FastAPI 基础后端

## 目标

这一阶段的目标非常简单：

> 先不要急着接模型，先把最基础的 HTTP 服务跑起来。

也就是先完成：

```text
用户请求 -> FastAPI 路由 -> 返回结果
```

---

## 项目结构初步成型

我逐步把目录整理成了这种形态：

```text
blog-agent/
├── app/
│   ├── routes/
│   │   └── agent.py
│   ├── services/
│   │   ├── llm_service.py
│   │   └── file_service.py
│   ├── config.py
│   └── main.py
├── outputs/
├── .env
├── .gitignore
└── requirements.txt
```

虽然这是后面逐渐稳定下来的结构，但核心思想在这一阶段已经出现了：

- `main.py`：应用入口
- `routes`：接口层
- `services`：业务逻辑层
- `config.py`：配置层

这说明项目不再是单文件脚本，而是一个有分层意识的后端工程。

---

## 最小可运行入口

我先写了一个最小的 FastAPI 入口，让服务先能启动。

核心思想是：

- 启动服务
- 提供一个 `/` 接口
- 提供一个 `/chat` 接口

---

## 第一版 `/chat`：回声测试接口

最开始的 `/chat` 并没有真正接入模型，而是一个测试版：

```python
@router.post("/chat")
def chat(data: ChatRequest):
    return {
        "reply": f"你刚刚说的是: {data.message}"
    }
```

---

## 为什么先做回声版，而不是直接接模型

因为这一步的目的不是做智能能力，而是验证链路。

我需要确认：

1. 接口能不能访问
2. JSON 请求体能不能正确解析
3. 服务能不能正常返回响应

所以它本质上是一个“后端管道联通测试”。

如果这一步都没通，后面接模型只会更混乱。

---

## 接口联调方式

我通过 `curl` 测试接口：

```bash
curl -X POST "http://127.0.0.1:8000/chat" \
-H "Content-Type: application/json" \
-d '{"message":"今天学了 Git 和 Docker"}'
```

返回：

```json
{"reply":"你刚刚说的是: 今天学了 Git 和 Docker"}
```

这说明第一阶段后端基础链路跑通了。

---

## 遇到的问题：端口占用

后面多次启动服务时我遇到了这个问题：

```bash
[Errno 48] Address already in use
```

### 原因分析

原因是另一个终端里已经有一个 `uvicorn` 进程在跑了，8000 或 8001 端口已经被占用。

### 解决方式

我通过：

```bash
lsof -i :8000
kill -9 PID
```

或者直接换端口：

```bash
uvicorn app.main:app --reload --port 8001
```

后来我也逐渐明白：

> 开发时如果已经有一个服务实例在运行，就不要重复启动；  
> 如果改了 `.env`，还必须重启进程，配置才会重新加载。

---

## 这一阶段的成果

这一阶段完成之后，项目已经具备：

- 基础 FastAPI 服务
- 可访问的 `/chat` 接口
- 请求 / 响应闭环
- 初步目录结构

虽然还没有真正的 AI 能力，但后端骨架已经搭起来了。

---

# 五、阶段三：接入 LLM，替换“回声回复”

## 这一阶段的目标

从这一步开始，项目要从：

```text
用户请求 -> 假回复
```

升级成：

```text
用户请求 -> LLM -> 真正生成内容
```

也就是说，接口不再只是重复用户输入，而是要调用模型来完成整理和生成。

---

## 配置管理：引入 `.env`

为了不把 API Key 和模型参数写死在代码里，我把这些内容放进了 `.env` 文件：

```env
OPENAI_API_KEY=你的key
OPENAI_BASE_URL=https://api.deepseek.com
OPENAI_MODEL=deepseek-chat
NOTES_DIR="/Users/wie0/Library/Mobile Documents/iCloud~md~obsidian/Documents/obsidian/wie0_vault/Execution"
```

这样后续要切模型、改目录，都可以通过配置完成，而不需要反复改代码。

---

## `config.py` 的作用

我在 `config.py` 中使用 `pydantic-settings` 读取环境变量。

这层的作用是：

- 集中管理配置
- 让代码能以 `settings.OPENAI_API_KEY` 的方式读取参数
- 避免硬编码

这一步是很典型的工程化处理，不仅让代码更清晰，也方便后续维护。

---

## 遇到的问题：配置字段不匹配

服务启动时我遇到了非常典型的报错：

```text
Extra inputs are not permitted
```

具体是：

- `.env` 里有 `OPENAI_BASE_URL`
- `.env` 里有 `OPENAI_MODEL`
- 但 `Settings` 类里没有定义对应字段

### 原因分析

`pydantic-settings` 会校验输入，如果环境变量出现了未定义字段，它会直接报错。

### 解决方式

我补全了 `Settings` 类里的字段定义，并设置：

```python
extra = "ignore"
```

或者在新版写法中使用对应配置项，让多余字段可以被忽略。

### 这一问题带来的理解

这件事让我很明确：

> 配置系统不是“写个环境变量就能用”，而是代码定义和环境变量必须保持一致。

---

## 封装 `llm_service.py`

我把模型调用逻辑放进了 `app/services/llm_service.py`。

它的职责是：

- 初始化模型客户端
- 接收用户输入
- 传入 Prompt
- 返回模型生成结果

这一层的好处是，接口层不需要知道模型怎么调，只要知道“调用这个函数就能拿到结果”。

这也是一个很重要的分层思想：

- 路由层只负责接请求和返回响应
- 服务层负责具体业务逻辑 

---

## Prompt 的作用在这一步开始真正体现

在 `llm_service.py` 里，我逐渐意识到：

> 真正决定“生成内容质量”的，不是接口，而是 Prompt。

比如我后面在 `system prompt` 中开始加入这些规则：

- 第一行必须是一级标题
- 标题不能带日期
- 不要使用 emoji、图标
- 使用正式、简洁的技术博客风格
- 尽量保留真实开发过程和问题分析

这些规则让生成结果开始从“普通回答”转向“可保存的技术文档”。

---

## 这一阶段的成果

这一步完成后，项目已经从一个测试接口，升级成了一个真正可以调用 LLM 的后端服务。

现在它已经具备：

- 模型接入能力
- 配置驱动能力
- Prompt 控制能力
- 路由层和服务层分离

---

# 六、阶段四：从“生成文本”到“写入 Markdown 文件”

## 这一阶段的目标

虽然接入 LLM 后，接口已经能返回比较完整的内容，但这还不够。

因为我的目标不是“拿到一段回复”，而是：

> 自动生成 Markdown 文件，并真正保存下来。

所以这一阶段的目标是完成：

```text
LLM 输出 -> 组装内容 -> 写入 .md 文件
```

---

## 一开始的问题：接口能返回内容，但没有文件

我最开始遇到的一个误区是：

接口已经返回了完整文段，我以为功能已经完成了。

但后来发现，`outputs/` 目录里根本没有生成文件。

### 原因分析

之前只是做到了：

```text
用户请求 -> LLM -> 返回 reply
```

但还没有做：

```text
reply -> 写入文件
```

也就是说，“生成内容”和“保存内容”是两件不同的事。

---

## `file_service.py` 的作用

为了解决这个问题，我增加了 `app/services/file_service.py`，专门负责：

- 创建目录 
- 生成文件路径
- 写入 Markdown
- 返回保存路径

这一步让项目第一次拥有了“产出物”。

这非常关键，因为它把 Agent 从“聊天接口”变成了“内容生产工具”。

---

## 路由层改造

这时候 `/chat` 的逻辑也变成了：

```text
接收输入 -> 调用 LLM -> 保存 Markdown -> 返回内容和路径
```

返回结构也开始不仅仅有 `reply`，还会有 `saved_to`。

这样我每次生成后，不仅能看到内容，还知道它被保存到了哪里。

---

## 这一阶段的成果

到这里，项目第一次形成闭环：

```text
用户输入 -> FastAPI -> LLM -> Markdown -> 文件写入
```

这意味着它已经不是“会回答问题”，而是“会产出文档”。

---

# 七、阶段五：输出到 Obsidian 笔记目录

## 为什么要做这一步

虽然之前已经能写入 Markdown 文件，但默认保存目录是项目本地的 `outputs/`。

这对测试来说没问题，但对我的真实使用场景来说不够自然。

我的目标是：

> 让生成结果直接进入我的 Obsidian 笔记系统。

这样我就可以把 Agent 直接嵌入到自己的学习记录流程中，而不是生成后还要手动搬运。

---

## 配置保存目录

我把路径放进 `.env`：

```env
NOTES_DIR="/Users/wie0/Library/Mobile Documents/iCloud~md~obsidian/Documents/obsidian/wie0_vault/Execution"
```

然后在 `file_service.py` 中改成读取 `settings.NOTES_DIR`。

---

## 为什么路径放 `.env` 更合理

如果我直接把路径写死在代码里，会有这些问题：

- 路径难以维护
- 换目录要改代码
- 换设备更麻烦
- 不适合提交到仓库中

所以正确方式是：

> 路径属于环境配置，应该放进 `.env`。

---

## 这一阶段的成果

这一阶段完成后，项目不再只是把内容写进项目目录，而是能直接进入我的 Obsidian 笔记仓库。

这使它真正开始服务我的实际学习和写作流程。

---

# 八、阶段六：文件名优化——日期、标题、双日期问题

## 初始问题：文件名固定

最开始保存时，我直接用了固定文件名，比如：

```text
learning-note.md
```

这会带来两个问题：

1. 每次生成都会覆盖旧文件
    
2. 文件名无法反映文章主题
    

---

## 优化方向：文件名自动带日期和主题

于是我把目标改成：

```text
日期 + 标题主题
```

例如：

```text
2026-03-22-Learning-Agent-项目初始化与核心功能搭建.md
```

这样每篇文章都有：

- 时间信息
    
- 主题信息
    
- 唯一性
    

---

## 技术实现思路

我在 `file_service.py` 中增加了这些逻辑：

1. 从 Markdown 第一行中提取一级标题
    
2. 清洗标题中的非法字符
    
3. 自动拼接当前日期
    
4. 生成最终文件名
    

---

## 后来又遇到的问题：双日期叠加

后来我发现生成出来的文件名长这样：

```text
2026-03-22-2024-05-15-Learning-Agent-项目初始化与核心功能搭建.md
```

### 原因分析

问题在于：

- 程序在文件名前面自动加了一次日期
    
- 模型生成的标题里又带了一次日期
    

结果就变成双日期叠加。

---

## 解决方式

我做了两层处理：

### 1. Prompt 限制标题不要带日期

在 `system prompt` 里明确要求：

- 标题只包含文章主题
    
- 不要包含日期
    

### 2. 文件名生成逻辑里增加“去日期清洗”

即使模型偶尔还是带日期，程序也会先把标题开头的日期去掉，再参与文件名拼接。

---

## 这一阶段的成果

这一轮优化之后，文件命名规则变得更稳定，也更适合长期积累技术博客和学习文档。

---

# 九、阶段七：Prompt 优化——真正控制内容质量

## 为什么 Prompt 优化重要

在这个项目推进过程中，我越来越明确一件事：

> 内容质量的上限，很大程度上取决于 Prompt。

也就是说，代码只能决定：

- 怎么调接口
    
- 怎么保存文件
    
- 怎么命名
    

但真正决定“写成什么样”的，是 Prompt。

---

## 我通过 Prompt 控制了哪些内容

目前我已经开始约束这些内容：

- 第一行必须是一级标题
    
- 标题不要带日期
    
- 不要用 emoji 和图标
    
- 风格偏技术博客
    
- 内容要具体，不要空泛
    
- 保留真实开发过程和问题分析
    
- 输出适合直接保存成 Markdown 博客草稿
    

---

## 这一步带来的变化

Prompt 优化后，生成结果开始明显从：

- 泛泛而谈的回答
    
- 普通总结
    
- 结构松散的文字
    

变成：

- 更像正式博客
    
- 更像技术记录
    
- 更适合长期存档
    

---

# 十、阶段八：从接口测试转向可用产品——增加网页输入页

## 为什么这一步很关键

虽然我已经完成了后端、模型、保存逻辑，但如果每次使用都要：

- 打开终端
    
- 写 `curl`
    
- 拼 JSON
    
- 粘贴长文本
    

那这个 Agent 很难真正成为日常工具。

所以我逐渐意识到：

# 十二、问题分析：今天踩过的坑与经验总结

## 1. `pip` 命令找不到

### 现象

执行：

```bash
pip install -r requirements.txt
```

报错：

```bash
zsh: command not found: pip
```

### 原因

macOS 环境下没有直接暴露 `pip` 命令，需要通过 `python3 -m pip` 使用。

### 解决

改用：

```bash
python3 -m pip install -r requirements.txt
```

### 经验

以后尽量统一使用：

```bash
python3 -m pip ...
```

更稳妥。

---

## 2. 端口占用

### 现象

启动 `uvicorn` 时报：

```bash
Address already in use
```

### 原因

另一个终端里的服务还在运行。
### 解决
- 关闭旧进程
- 或更换端口
### 经验

开发时要清楚当前哪个服务实例在运行，修改 `.env` 后记得重启。

---

## 3. `.env` 配置字段不匹配

### 现象

启动时报 `Extra inputs are not permitted`

### 原因

`.env` 中定义了变量，但 `Settings` 类中没有对应字段。

### 解决

在 `config.py` 中补齐字段定义，并处理额外字段。

### 经验

配置层和代码层必须同步，不然启动阶段就会崩。

---

## 4. 生成内容但没有文件

### 现象

接口返回了文段，但目录里没有生成文件。

### 原因

只有 LLM 生成，没有调用保存逻辑。

### 解决

增加 `file_service.py`，在路由中调用 `save_markdown()`。

### 经验

返回内容和副作用是两回事，不能只看接口响应。

---

## 5. 文件名固定

### 现象

所有文件都叫同一个名字。

### 原因

调用 `save_markdown()` 时仍然传入固定字符串。

### 解决

让文件名生成逻辑自动根据日期和标题处理，不再手动写死。

### 经验

改函数实现时，也要同步检查调用方有没有继续使用旧逻辑。

---

## 6. 双日期问题

### 现象

文件名带两个日期。

### 原因

程序自动加日期，模型标题里也带日期。

### 解决

Prompt 限制标题不带日期 + 程序清洗标题日期。

### 经验

模型输出是不稳定的，程序侧最好做兜底处理。

---

# 十三、下一步优化方向

到这里，项目已经跑通第一版闭环，但还有很多值得继续推进的方向。

## 1. Prompt 精修

进一步固定输出风格，让内容更像正式技术博客，而不是普通总结。

## 2. 结构模板化

比如固定输出：

- 摘要
    
- 正文
    
- 问题分析
    
- 当前成果
    
- 下一步计划
    
- 标签
    

这样生成结果会更稳定。

## 3. 多模式支持

后续可以支持不同模式：

- 技术博客模式
    
- 学习日报模式
    
- 问题复盘模式
    

## 4. 多轮交互

当前偏单轮生成，后续可以做成：

- 补充细节
    
- 继续扩写
    
- 改写成更正式版本
    
- 生成摘要版
    

## 5. 与博客系统进一步整合

不仅写入 Obsidian，也可以进一步输出到博客文章目录，自动生成 frontmatter，真正成为博客写作流水线的一部分。

## 6. 前端继续完善

把现在的最小页面进一步升级成更好用的输入工具，让它成为一个真正可持续使用的小产品。
