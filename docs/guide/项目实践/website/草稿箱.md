# 博客部署与环境配置过程中的问题记录

在博客同步、部署和环境配置的过程中，遇到了一些比较典型的问题。虽然最后都解决了，但这些问题本身很值得记录下来，因为它们反映了我在搭建博客时踩过的坑，也能帮助我以后少走弯路。

---
## 一、`docker pull nginx` 失败的问题

### 1. 问题现象

在部署博客时，我需要先拉取 `nginx` 镜像，但是执行：

```bash
docker pull nginx
```

时失败了。

### 2. 一开始的判断

刚开始我以为是 **CPU 架构问题**，比如镜像和本机架构不匹配，所以一直往这个方向排查。

后来查了些资料，又结合实际情况看下来，发现这类问题**大概率不是架构本身导致的**，而更可能是：

- Docker 环境没有装好
    
- 网络访问 Docker 镜像仓库太慢
    
- 国内访问 Docker Hub 不稳定
    
- 没有配置镜像加速
    

### 3. 最终原因

最终更符合实际的原因是：**网络问题**。

也就是说，不是镜像不能用，而是拉取镜像的过程卡在了访问速度上，导致一直失败或者超时。

### 4. 解决办法

我后来采用了比较常见的做法：**先配置 Docker 镜像加速，再重新拉取镜像**。

典型思路是：

1. 配置加速器
2. 重启 Docker
3. 再执行 `docker pull nginx`

### 5. 这个问题的本质

这个问题本质上不是镜像本身坏了，而是：

> Docker 默认要访问的镜像源在当前网络环境下太慢，导致拉取失败。

### 6. 经验总结

这件事让我意识到，很多时候不是“命令错了”，而是“网络环境不适合默认流程”。  
以后遇到 Docker 拉镜像失败，第一反应应该先查：

- Docker 是否安装正常
- 网络是否通
- 是否需要镜像加速
- 是否能正常访问 Docker Hub

## 二、同步上传博客时，自动化脚本卡住的问题

### 1. 问题现象

在把博客内容同步上传到服务器的时候，我写了一个自动化脚本，本来是想减少重复操作的，但在上传到服务器这一步卡住了。

### 2. 具体卡点

主要有几个问题：

#### （1）电脑没有生成 SSH Key

也就是说，本地机器还没有配置好用于 SSH 登录的密钥。

#### （2）密钥粘贴不完整

在配置密钥时，公钥或私钥内容没有完整写入，导致认证失败。

#### （3）密钥私钥没有被正确识别

可能是本地 SSH 配置没有配好，或者相关文件里有两行指令被注释掉了，导致系统没有按预期读取密钥。

### 3. 这个问题的本质

这个问题的核心不是“脚本本身错了”，而是：

> 自动化脚本依赖 SSH 免密登录，但 SSH 密钥和认证流程没有提前配置正确。

换句话说，脚本要想顺利上传文件，前提是本地到服务器的认证链路必须先打通。

### 4. 解决思路

解决这类问题，通常要按顺序检查：

1. 本地是否已经生成 SSH key
2. 公钥是否已经正确放到目标服务器或 GitHub 平台
3. 私钥是否在本地正确保存
4. `~/.ssh/config` 之类的配置是否正确
5. SSH 连接是否能手动连通

### 5. 经验总结

这次让我明白，自动化脚本不是“写出来就能跑”，它依赖很多前置条件。  
如果认证没配好，脚本再完整也会卡在上传这一步。

## 三、安装 Node.js 和 Hexo CLI 时下载很慢的问题

### 1. 安装流程回顾

我先按照官方流程安装了 `nvm`（Node Version Manager），然后再通过 `nvm` 安装了 `Node.js`。  
官方安装入口我当时是按 Node.js 的标准安装流程来的。

安装完 Node.js 后，我又继续安装 `Hexo CLI`：

```bash
npm install -g hexo-cli
```

### 2. 遇到的问题

在安装 Hexo CLI 的时候，`npm` 一直卡在默认源下载，速度非常慢。

也就是说，命令不是完全报错，而是：

- 下载很慢
- 安装时间很长
- 有时像卡住一样

### 3. 解决办法

我最后采用的是：**先把 npm 镜像切换到国内源，再重新安装**。

操作步骤如下：

#### 先停止当前安装

```bash
Ctrl + C
```

#### 切换 npm 镜像源

```bash
npm config set registry https://registry.npmmirror.com
```

#### 确认镜像是否切换成功

```bash
npm config get registry
```

正常情况下应该输出：

```bash
https://registry.npmmirror.com/
```

#### 重新安装 Hexo CLI

```bash
npm install -g hexo-cli
```

如果还是有权限问题，可以再加：

```bash
sudo npm install -g hexo-cli
```

### 4. 这个问题的本质

这个问题本质上不是 Hexo 有问题，而是：

> npm 默认源在当前网络环境下速度太慢，导致全局安装过程被拖住。

### 5. 为什么切镜像能解决

因为国内镜像源距离更近、访问更快，下载 npm 包时延迟更低，安装自然就稳定很多。

### 6. 经验总结

这件事说明，开发环境里最容易被忽略的，不一定是代码，而是：

- 包管理器源
- 网络环境
- 安装链路
- 权限问题

很多时候，一个“安装很慢”的问题，背后只是因为默认源不适合当前网络环境。

## 四、这几类问题的共同规律

这几个问题虽然表面上不同，但本质上很像，都是**环境配置问题**，不是业务逻辑问题。

### 共同点有三个：

1. **不是代码本身坏了，而是前置条件没准备好**
    - Docker 镜像拉取失败
    - SSH Key 没配好        
    - npm 默认源太慢
2. **都属于“工具链问题”**
    - Docker
    - SSH        
    - npm / nvm / Node.js
    - Hexo CLI
3. **都说明部署不是单一步骤，而是一个完整链路**
    - 本地环境
    - 认证环境
    - 网络环境
    - 服务器环境
    - 容器环境

---

## 五、整理后的经验结论

通过这几个问题，我对博客部署这件事的理解更完整了：

- **Docker 镜像拉取失败**，先看网络和镜像源，不要先怀疑架构。
- **自动化脚本卡住**，先检查 SSH Key 和认证链路。
- **npm 安装过慢**，优先切换国内镜像源。
- **部署不是一条命令，而是一整套环境准备流程**。

---

## 总结

> 在博客部署和环境配置过程中，最开始主要遇到的是 Docker 拉取镜像失败、SSH Key 未正确配置、npm 默认源下载过慢等问题。  
> 这些问题表面上分别属于容器、认证和包管理器，但本质上都属于开发环境的前置条件没有准备好。  
> 通过配置 Docker 镜像加速、补全 SSH Key、切换 npm 国内镜像源，这些问题最终都得到了处理，也让我对整个博客部署链路有了更清晰的理解。
