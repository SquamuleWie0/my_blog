# 从 Hexo 迁移到 VitePress的博客重构与部署流程记录

> 在整理这个博客架构的时候，我一直没能调整到理想状态。GitHub 上看了一些模板，感觉都不太符合我的使用习惯，尤其是目录结构比较混乱，发现其实我自己最开始的规划就不够清晰。
>
> 后来我发现VitePress很符合我的想要的架构，就开始尝试把博客从 `Hexo` 迁移到了 `VitePress` 上，并重新梳理了目录结构、内容组织方式和部署流程。

---

## 一、迁移前的整体思路

这次迁移不是单纯换一个主题，而是把整个博客从“Hexo 博客站点”调整成“VitePress 文档站点”。

### 这次迁移要解决的核心问题

1. 原来的目录结构不清晰，文章分类混乱。
2. Hexo 的一些语法和自动化方式不适合现在的内容管理方式。
3. 我希望博客更像一个结构化知识库，而不是一个单纯的静态博客。
4. 我希望后续能更方便地管理笔记、项目和技术学习内容。

### 迁移后的目标

- 用 VitePress 作为博客主体框架。
- 让文章按照目录分类清晰展示。
- 让导航栏、侧边栏、文章路径统一规范。
- 让本地预览和服务器部署流程更稳定。

---

## 二、迁移前准备：先把结构想清楚

迁移之前，最重要的事情先规划内容组织方式，构思清楚呈现出来的效果会更满意。

### 1. 明确博客内容结构

我先把博客内容分成几类：

- 工程工具
- 基础知识
- 技术学习
- 课程笔记
- 项目实践

我照搬了obsidian里面部分文件树的结构，目的是为了每一类内容都有固定位置，后面不管是写文章还是配导航，都会比较清楚。
但是按照后续的规划这样划分其实并不是最能展现我产出的内容的设计，所以现在暂时这样划分，再根据后续的输出进行调整。

### 2. 梳理原 Hexo 内容

Hexo 的文章主要放在 `_posts` 里，但那个结构更偏博客发布，不太适合我现在这种“笔记仓库 + 技术文档”的形式。

所以我需要先确认：
- 哪些文章要保留
- 哪些文章要迁移到新目录
- 哪些静态资源还要继续使用
- 哪些内容需要重新整理
### 3. 规划 VitePress 目录结构

VitePress 的核心目录主要有三个：

```text
docs/
├─ guide/        # 博客正文内容
├─ .vitepress/    # 配置文件、主题、样式等
└─ public/        # 静态资源
```

### 这三个目录分别干什么

- `docs/guide/`：放所有正文文章。
- `docs/.vitepress/`：放网站配置文件，比如导航栏、侧边栏、主题样式。
- `docs/public/`：放图片、附件、视频等静态资源。
### 这个阶段我实际想实现的

这几个文件夹使用的核心是在给整个博客定规则：

- 文章放哪里
- 页面怎么跳转
- 分类怎么展示
- 图片怎么引用
- 后面怎么维护

## 三、迁移文章和资源：把 Hexo 内容搬到 VitePress

这一阶段主要是把原来 Hexo 的文章内容，整理到 VitePress 能识别的结构里。

### 1. 迁移文章

我把 `Hexo` 的文章移动到了 `docs/guide` 下的对应分类目录。

这样做的作用是：
- 首先保证文章按照主题分类存放
- 也使得我的侧边栏更容易配置
- 路径和页面结构保持一致

### 2. 调整文件夹层级

这里的“调整文件夹层级”，本质上是在管理博客内容的逻辑结构。

比如：

```text
docs/guide/基础知识/计算机网络/计算机网络基础.md
```

这不是随便放文件，而是在告诉博客：

- 这篇文章属于“基础知识”
- 下面还有一个子分类“计算机网络”
- 文章本身是“计算机网络基础”

这样做之后，后面配侧边栏的时候就很自然。（其实就是我笔记的存放逻辑）

### 3. 清理 Hexo 特有语法

Hexo 和 VitePress 的文章渲染逻辑不一样，所以有些 Hexo 的特殊语法不能直接带过去。

比如：
- `{: .class}`
- 某些插件短代码
- 自动插入的标签语法
- 一些不兼容的图片或组件写法

这些内容如果留在 VitePress 里，有可能导致：
- 页面报错
- 构建失败
- 渲染异常

所以一定要把这些不兼容内容清理掉，或者用通用标准 Markdown 重新写。

### 4. 统一 front-matter

`front-matter` 是 Markdown 文件开头那段 YAML 配置块，用来描述文章信息。

例如：

```yaml
---
title: 文章标题
categories: [一级分类, 二级分类]
tags: [标签1, 标签2]
---
```

### 这个配置是干什么的

它的作用是告诉 VitePress：

- 这篇文章叫什么名字
- 属于哪个分类
- 可以怎么归档和展示

### 为什么要统一它

因为我之前在 Hexo 里用的文章格式比较杂，有些文章有属性，有些没有，有些还加过脚本自动写入的内容，导致现在 Obsidian 和博客内容不够统一。

### 这一步的重要作用

- 保证每篇文章格式一致
- 方便后续脚本处理（这个很重要）
- 方便侧边栏和导航组织
- 减少构建错误
### 5. 迁移静态资源

`Hexo `的静态资源通常放在 `source/` 里，但 VitePress 更推荐统一放到 `docs/public/`。

这样做的好处是：

- 图片引用更稳定
- 页面构建更容易
- 后续部署更省心

文章里引用图片时，统一写成：

```markdown
![示例图片](/img/example.png)
```

## 四、配置 VitePress：让页面真正“活起来”

>这一阶段主要是在 `docs/.vitepress/` 里修改配置文件，让博客能够正常显示、跳转和渲染。
### 1. 打开 `docs/.vitepress/config.js`

这个文件是整个 VitePress 网站的总配置文件，相当于博客的“总控制台”。

它负责什么
- 网站标题
- 网站描述
- 顶部导航栏
- 侧边栏
- 编辑链接
- 页面行为配置

它决定了：

- 浏览器标签页显示什么
- 顶部导航栏点到哪里
- 左边目录树怎么显示
- 是否显示最后更新时间
- 是否提供 GitHub 编辑入口
---

### 2. 打开 `docs/.vitepress/sidebar.js`

>这个文件是侧边栏配置文件。
#### 它负责什么

它决定每一类文章在左边目录里怎么展示。

比如：

- 基础知识
- 工程工具
- 技术学习
- 课程笔记
- 项目实践

#### 为什么要单独分出来

因为目录层级比较复杂，如果全部写在 `config.js` 里会很乱，单独拆一个 `sidebar.js`，后期更容易维护。这也是我最满意的效果。

---
### 3. 自定义样式文件

如果页面样式不符合预期，可以去这些地方改：
- `docs/.vitepress/theme/index.js`
- `docs/.vitepress/custom.css`
#### 它们分别干什么

`theme/index.js`

这个文件适合做更深层次的主题定制，比如：

- 替换默认组件
- 调整页面布局
- 自定义首页模块

`custom.css`

这个文件适合做样式调整，比如：

- 字体大小
- 间距
- 首页 Hero 布局
- 颜色、边距、对齐方式
#### 我之前调整样式时遇到的问题

- 有些 CSS 不生效
- 主题默认样式优先级比较高
- hero 布局调了半天效果不明显

所以后面我更倾向于先在 `custom.css` 里小范围修改，避免直接改动太多主题结构，导致出错。

## 五、本地预览与调试

>先确认网页能正常跑起来
#### 打开本地开发服务

```bash
npx vitepress dev docs
```
### 这个命令是干什么的

它会在本地启动一个预览服务器，让我先在本机查看整个博客的效果。

### 这一步要检查什么

- 首页是否正常显示
- 顶部导航栏是否能点
- 侧边栏是否正确展开
- 文章内容是否能打开
- 图片和链接是否正常
- 自定义样式是否生效
### 为什么这一步很重要

因为本地预览没问题，才值得继续 build 和部署。否则直接上传服务器，出问题会更难排查。

## 六、构建静态文件

>把 Markdown 变成可部署网页
#### 执行构建命令
```bash
npx vitepress build docs
```

#### 这个命令干什么

它会把：

- Markdown 文章
- 配置文件
- 样式
- 页面结构

一起编译成一套静态网页文件，也就是我们在网页上访问的东西。
#### 构建完成后输出在哪里

```text
docs/.vitepress/dist/
```

#### 这个目录是什么

这是最终可以部署到服务器上的网站文件夹。

里面通常会有：
- `index.html`
- `assets/`
- 各类页面文件
- 静态资源文件

这一步是博客不再只是 Markdown 文件，而是成为一个能部署的网站的关键。

## 八、这次迁移过程中遇到的问题

### 1. 目录结构混乱

这是最早遇到的问题。

**问题本质**
Hexo 的目录结构和 VitePress 的目录结构不同，我一开始没有把“文件夹 = 分类 = 路径”这个逻辑理顺，把自己都弄的乱七八糟的。

这就导致了页面跳转混乱、侧边栏难配、文章归类不清晰。

**最终处理方式**
重新规划目录结构、 把文章按主题重新归档、 配合 `sidebar.js` 统一管理

---
### 2. front-matter 格式不统一

**问题本质**
之前有些文章带了脚本自动生成的属性，有些文章没有，导致格式不一致。
就导致我的页面信息不统一、构建和渲染不稳定、维护成本高很麻烦。

先是尝试着逐步清理旧属性，后面直接用脚本批量修正了，就好了。

---
### 3. 图片和静态资源路径错误

**问题本质**：
VitePress 构建时会解析资源路径，`/img/xxx` 会被当作站点根路径资源，如果文件不在 `public/` 中，build会直接报错。

我把图片统一放到 `public`，将所有的图片文件路径统一管理，不再混用 Obsidian 的内部图片语法就好了。

---
### 4. 导航栏和路径不匹配

**问题本质**：
我最开始把导航栏直接指向目录，但目录里没有对应入口页面，所以404了。
直接导致我这个顶部导航失效，页面访问不到。

我先是直接让导航直接跳文章我没做成功，其实也可以给每个目录补入口页，但是最后我直接按实际文章路径来配的就能成功跳转了。

---
### 5. Markdown 没有大标题

**问题本质**：
我以前在 Obsidian 里写笔记时，很多文章正文本身没有 `# 一级标题`。
页面结构不清晰，标题层级不完整，看着怪怪的。

后面让ai写了个用脚本自动加标题
