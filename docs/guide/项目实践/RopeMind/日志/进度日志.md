**2026-05-06**

今天重点看懂了 `langchain_llm.py` 和 `langgraph_agent.py` 两个核心文件，对主流 Agent 框架的理解清晰了很多。

`langchain_llm.py` 理解了 LangChain 的基础作用：它不是让模型变聪明，而是用更标准、更流程化的方式组织模型调用。原来手写的 `ask_deepseek(prompt)`，现在变成了 `ChatPromptTemplate → llm → response.content` 这样的链式流程。`prompt_template | llm` 本质上就是把“提示词模板”和“大模型调用”串成一条流水线，让模型调用更规范，也方便后续接入检索、工具和 LangGraph。

`langgraph_agent.py` 理解了 LangGraph 的核心：Agent 不是一大坨函数，而是一组节点不断读写同一个状态。每个节点函数负责一个环节，比如路由、检索、评估、回答。节点接收当前 `state`，执行自己的任务，然后返回更新后的 `state`。返回值里的 `**state` 用来保留已有信息，后面的字段就是这个节点新增或修改的信息。

现在对 Agent 流程的理解更具体了：用户问题进入后，不是直接交给模型回答，而是先进入 `route` 节点判断是否需要检索；需要的话进入 `retrieve` 节点获取上下文；再进入 `evaluate` 节点判断上下文是否足够；最后进入 `answer` 节点生成回答。整个过程通过同一个 `state` 串起来，LangGraph 负责决定节点执行顺序。

LangChain 负责标准化模型调用，LangGraph 负责组织多步骤流程，`state` 负责在各节点之间传递信息。这对我后续掌握主流 Agent 工程能力很重要。

今天读完了 `chroma_indexer.py`，这个部分主要是 RAG 的基本链路。首先筛选有效文件，把每个有效文件读取成 `Document`，其中包含文本内容和 metadata，并保存到 `documents` 列表。然后把较长的 `Document` 切成适合检索的 chunk。接着创建 embedding 模型，把每个 chunk 转换成向量。Chroma 会保存 chunk 文本、metadata 和对应的 embedding 向量，并基于这些向量建立可检索的索引。Chroma 索引第一次使用时需要创建，后续可以直接加载。进行语义搜索时，用户问题也会被向量化，然后通过 `similarity_search()` 和已有 chunk 向量进行相似度匹配，返回最相关的 top_k 个片段，最后把这些片段拼接成上下文返回给 Agent 使用。

`api.py `是 RepoMind 的 FastAPI 后端入口，主要负责接收前端请求、调用业务逻辑，并把处理结果返回给前端。

FastAPI 应用对象 `app` 用来注册接口。每个接口本质上由三部分组成：

一个 URL + 一个 HTTP 方法 + 一个 Python 函数

当前端请求某个接口时，FastAPI 会根据 URL 和 HTTP 方法找到对应函数，并自动解析前端传来的参数。接口函数内部通常会先做参数校验和错误处理，然后准备项目上下文、记忆上下文等对象，再调用核心业务逻辑函数，比如项目分析、LangGraph Agent 问答等。

业务逻辑执行完成后，接口函数 `return` 一个 Python dict。FastAPI 会自动把 `dict` 转成 JSON，并生成 HTTP 响应返回给前端。

现在的方向逐步向 LangChain / LangGraph 生态靠近。

对于标准 Agentic RAG，核心在于让Agent参与决策、工具调用、评估检测结果、并且根据结果重新调整流程。他的侧重在于Agent如果围绕着整个检索过程做决策。

标准 Agentic RAG 不是指某一个单独工具，而是指一种主流 Agent + RAG 架构：LLM 负责判断是否需要检索，retriever 被封装成 tool，Agent 通过 tool_call 调用检索工具，随后对检索结果进行相关性评估，并根据结果选择生成答案或改写问题重试。它的重点是让检索成为 Agent 流程中可决策、可评估、可重试的工具环节。RepoMind 后续应该在保持显式状态清晰的基础上，逐步向 LangChain / LangGraph 的 tool calling、ToolNode、structured output 等标准机制靠近。

Tool Calling 标准路由
官方流程是：

```
LLM 判断
↓
模型生成 tool_call
↓
tools_condition 检查有没有 tool_call
↓
ToolNode 执行对应 tool
↓
工具结果作为 ToolMessage 写回 messages

@tool：把普通函数变成 Agent 工具
bind_tools：把工具绑定给 LLM
tool_call：LLM 发出的工具调用请求
ToolNode：执行工具调用
tools_condition：判断下一步走工具还是结束
```

Tool Calling 是模型调用工具的标准流程；LLM 负责决定是否发起 tool_call，tools_condition 只负责检查有没有 tool_call，ToolNode 负责真正执行工具
bind_tools 是“给模型工具选项”；
tool_calls 是“模型这次实际发出的工具调用请求”；
AIMessage 是“模型某一次输出”，会被追加到 messages 列表里。

在 LLM 应用里，约束通常来自三层：
1. Prompt：告诉模型任务、规则、输出期望
	1.` Tool schema` / `Pydantic schema`：告诉模型工具参数或结构化输出格式
2. 框架机制：比如 `bind_tools`、`ToolNode`、`with_structured_output`

> `schema` 是数据结构规则；`with_structured_output(GradeDocuments)` 是把这个规则交给模型和 LangChain，让模型按这个格式输出，并让代码可以像访问对象字段一样读取结果。

在LangGraph中，节点函数必须是要满足传入state，返回state的。所以对于工具函数（str->str）不能直接注册成节点，需要 `ToolNode` 这个适配器。

**闭包（closure）**：
外层函数先接收额外参数，把它“装进背包”里，然后返回一个只需要 `state` 的内层节点函数；后面 LangGraph 正常调用这个节点时，节点还能使用背包里的 `chroma_tool`。
