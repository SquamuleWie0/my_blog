RepoMind 项目面试复盘稿

一、项目一句话介绍

RepoMind 是一个面向代码仓库理解与开发辅助场景的 Agentic RAG 工作台。

用户输入 GitHub 仓库地址后，系统会自动克隆仓库、扫描项目结构、生成 AI 项目导读、构建 Chroma 语义索引，并通过 LangGraph Tool Calling Agent 支持围绕项目进行多轮源码问答、实现定位、架构理解、项目历史恢复和持续追问。

它不仅可以用于从 0 理解陌生开源项目，也可以用于复盘自己的项目、辅助继续开发前的上下文恢复，以及作为代码仓库理解类任务的 Agentic RAG 工程实践。

二、项目解决什么问题

这个项目主要解决的是：如何让用户快速理解一个代码仓库，并围绕这个项目持续追问。

传统方式下，用户需要自己看 README、目录结构、源码文件、配置文件，理解成本比较高。RepoMind 的目标是把一个 GitHub 仓库转化成一个可对话、可追溯、可持续使用的项目知识工作台。

它支持：

1. 自动分析 GitHub 仓库
2. 生成 AI 项目导读
3. 构建源码语义索引
4. 围绕源码进行多轮问答
5. 输出参考文件来源
6. 保存项目历史和问答记录
7. 切换历史项目并继续追问

三、整体流程怎么走

1. 用户输入 GitHub URL。
2. 后端 /analyze 接口克隆仓库。
3. 系统扫描项目文件和目录结构。
4. 读取关键文件，比如 README、配置文件、核心源码文件。
5. repo_analyzer 生成项目基础分析结果。
6. llm_analyzer 生成 AI 项目导读。
7. chroma_indexer 对源码、README、配置文件进行 chunk 切分、metadata 标注和 Chroma 向量索引构建。
8. project_store 将项目地址、项目名称、技术栈、repo_path、file_tree、analysis_result、ai_report、basic_report 保存到 SQLite。
9. 前端展示项目导读。
10. 用户提问时，进入 LangGraph Agentic RAG 流程。
11. Agent 根据问题决定是否调用 Chroma 检索工具。
12. 系统收集检索上下文、评估检索结果，如果不相关就重写问题再检索。
13. 最后基于源码上下文生成回答，并返回参考文件。
14. 问答记录保存到数据库，后续可以从历史项目继续追问。

四、/analyze 接口做了什么

/analyze 是项目分析入口。

主要流程是：

1. 接收 repo_url。
2. clone_repo 克隆 GitHub 仓库。
3. scan_and_build_tree 构建项目目录结构。
4. scan_repo 扫描项目文件。
5. read_key_files 读取关键文件内容。
6. analyze_repo 分析项目类型、技术栈、关键文件等。
7. generate_markdown_report 生成基础报告。
8. generate_llm_report 生成 AI 项目导读。
9. build_chroma_index 构建 Chroma 源码语义索引。
10. save_project 将项目分析结果保存到 SQLite。
11. 更新 CURRENT_PROJECT，方便当前会话继续提问。
12. 返回 project_id、project_name、tech_stack、ai_report 和 chat_history。

五、/ask-v2 接口做了什么

/ask-v2 是当前项目的问答入口。

流程是：

1. 接收用户问题。
2. 检查当前是否已经分析过项目。
3. 从 memory_manager 构造 memory_context。
4. 调用 answer_with_langgraph。
5. LangGraph 根据问题进入 entry_router。
6. 如果问题是在问项目导读，就走 report_answer。
7. 如果是源码、架构、API、函数、数据库等问题，就进入 Agent Tool Calling 流程。
8. Agent 可以调用 Chroma 检索工具。
9. 系统对检索结果进行评估。
10. 如果检索结果不足，就 rewrite_question 后重新进入 Agent。
11. 最后 answer_node 生成答案。
12. 保存到 memory。
13. 如果有 project_id，也把问答保存到 project_messages 表。

六、历史项目继续追问怎么实现

历史项目继续追问走的是：

POST /projects/{project_id}/ask

流程是：

1. 根据 project_id 从数据库读取项目记录。
2. 读取 repo_path、file_tree、analysis_result、ai_report。
3. 读取该项目最近若干轮历史问答。
4. 拼接成 memory_context。
5. 调用 answer_with_langgraph。
6. Agent 基于恢复出来的项目上下文继续回答。
7. 回答完成后，把新问题和新答案写入 project_messages。
8. 前端重新渲染该项目完整历史问答。

这让系统不再依赖单次内存状态，而是可以围绕历史项目持续使用。

七、LangGraph Agentic RAG 是怎么设计的

当前主流程是：

entry_router
→ agent
→ tools
→ collect_context
→ grade_documents
→ answer 或 rewrite_question

更具体是：

entry_router
├─ report_answer → END
└─ agent
    ↓
    tools_condition
    ├─ no tool_call → direct_answer → END
    └─ tool_call → tools
                  ↓
              collect_context
                  ↓
              grade_documents
              ├─ relevant → answer → END
              └─ not relevant → rewrite_question → agent

每个节点职责：

1. entry_router_node
   判断用户问题是问项目导读，还是进入普通源码问答。

2. report_answer_node
   直接基于 project_report 回答项目导读相关问题，不触发 Chroma 检索。

3. agent_node
   调用绑定工具后的 LLM，让模型自主判断是否需要检索源码。

4. ToolNode
   执行 chroma_search_tool。

5. collect_tool_context_node
   从 ToolMessage 中提取检索上下文、sources 和 chunks。

6. grade_documents_node
   判断检索结果是否足够回答用户问题。

7. rewrite_question_node
   如果检索结果不相关，改写问题后重新进入 agent。

8. answer_node
   基于 retrieved_context、memory_context、project_report 等上下文生成最终回答。

9. direct_answer_node
   当模型没有调用工具时，直接回答。

八、Agent 相关可以按四层来讲

1. Tool Layer

把 Chroma 检索封装成 chroma_search_tool，让 Agent 可以通过 Tool Calling 检索源码上下文。

2. Workflow Layer

使用 LangGraph StateGraph 编排 entry_router、agent、tools、collect_context、grade_documents、rewrite_question、answer 等节点。

3. Context Layer

区分 project_report、memory_context、retrieved_context。

project_report 是项目级长期上下文，来自 /analyze 生成的 AI 项目导读。

memory_context 是对话级上下文，来自历史问答。

retrieved_context 是本轮问题通过 Chroma 检索得到的源码上下文。

4. Control Layer

通过 entry_router、grade_documents、rewrite_question 控制 Agent 行为。

entry_router 决定是否走项目导读回答。

grade_documents 决定检索结果是否可用。

rewrite_question 决定是否需要重写问题后再次检索。

九、为什么要做 project_report 路由

一开始项目导读只是展示在前端，Agent 并不知道用户说的“项目导读”“上面的报告”“刚才生成的报告”指的是后端的 ai_report。

所以当用户问“项目导读内容是什么”时，Agent 可能会去 Chroma 检索代码，把问题理解成 README 或 main.py 里的介绍内容。

为了解决这个问题，我把 /analyze 生成的 AI 项目导读作为 project_report 独立传入 LangGraph，并在 entry_router 中判断用户是否在问项目导读。如果是，就直接走 report_answer_node，不触发 Chroma 检索。

这样做的价值是：

1. 让项目导读成为明确的上下文源。
2. 避免报告问题误触发源码检索。
3. 让 Agent 能围绕 AI 导读继续解释和追问。
4. 体现了上下文分层管理能力。

十、为什么要做 Source Citation

RAG 回答最大的问题之一是模型可能编造来源文件。

所以我在 Chroma 检索阶段保留 metadata，从每个 chunk 中提取 source 文件路径，并在最终回答中输出参考文件。

这样用户可以知道答案依据来自哪些真实文件，例如：

参考文件：
- main.py
- app/database.py
- app/models.py

价值是：

1. 提升回答可追溯性。
2. 降低文件来源幻觉。
3. 方便用户继续打开对应文件验证。
4. 更适合代码仓库理解场景。

十一、为什么要做 grade_documents

不是每次检索结果都能回答用户问题。如果检索结果不相关，直接让模型基于错误上下文回答，会导致答非所问。

所以我设计了 grade_documents_node，让 LLM 判断 retrieved_context 是否足够回答用户问题。

如果相关，就进入 answer。

如果不相关，就进入 rewrite_question。

价值是：

1. 避免拿无关上下文强行回答。
2. 给 Agent 增加自我反思能力。
3. 为问题重写提供依据。
4. 提升复杂问题和模糊问题下的稳定性。

十二、为什么要做 rewrite_question

用户的问题有时比较口语化、模糊或者缺少关键词，第一次检索可能搜不到相关代码。

所以当 grade_documents 判断检索结果不相关时，系统会根据：

1. original_question
2. 当前问题
3. grade_reason
4. retrieved_context 摘要

生成一个更适合检索的 rewritten_question，然后重新进入 agent。

为避免无限循环，我设置了 rewrite_count，目前最多重写一次。

十三、数据库为什么要接入

一开始系统只用 CURRENT_PROJECT 保存在内存里。这样有几个问题：

1. 刷新页面后项目状态容易丢失。
2. 只能维护当前一个项目。
3. 无法查看历史分析记录。
4. 无法围绕历史项目继续追问。
5. 项目还停留在单次 Demo 状态。

所以我接入 SQLite + SQLAlchemy，把项目分析结果和问答记录持久化下来，让系统从单次 Agentic RAG Demo 升级成项目历史工作台。

十四、数据库设计

目前主要有两张表：

1. projects

保存每个分析过的仓库。

字段包括：

- id
- repo_url
- project_name
- tech_stack
- repo_path
- file_tree
- analysis_result_json
- ai_report
- basic_report
- created_at
- updated_at

2. project_messages

保存每个项目下的历史问答。

字段包括：

- id
- project_id
- question
- answer
- timestamp

project_messages.project_id 是外键，关联 projects.id。

这样一个项目可以对应多轮问答，切换项目时只加载当前项目对应的 messages。

十五、为什么用 SQLite

因为当前阶段是本地 MVP 和个人工作台场景，SQLite 简单、轻量、无需额外部署数据库服务，适合快速验证项目历史持久化能力。

后续如果做用户注册登录、多用户隔离和服务器部署，可以迁移到 PostgreSQL。

十六、为什么用 SQLAlchemy

SQLAlchemy 提供 ORM 能力，可以用 Python 类来定义表结构和操作数据库，比直接写 SQL 更适合和 FastAPI 项目集成。

它也方便后续扩展，比如：

1. 增加 users 表。
2. 给 projects 增加 user_id。
3. 增加 agent_runs 表。
4. 保存 tool_calls、sources、grade、rewrite 等结构化执行轨迹。
5. 后续迁移到 PostgreSQL。

十七、project_store.py 主要做什么

project_store.py 是数据库操作封装层。

主要函数：

1. save_project

保存或更新项目记录。如果 repo_url 已存在，就更新项目导读、分析结果和 updated_at；如果不存在，就新增项目。

2. add_message

保存一轮用户问题和 Agent 回答。

3. list_projects

返回历史项目列表，用于左侧历史项目栏。

4. get_project

根据 project_id 获取项目详情，包括 ai_report、file_tree、analysis_result、repo_path 等。

5. list_messages

获取某个项目的历史问答记录。

十八、前端工作台做了什么

前端从原来的单页 Demo，升级成了工作台形态。

当前支持：

1. 左侧历史项目栏。
2. 点击历史项目切换当前项目。
3. 恢复 AI 项目导读。
4. 恢复历史问答。
5. 项目导读 / 项目对话 Tab 切换。
6. 围绕历史项目继续提问。
7. 提问后将新回答保存到对应项目历史中。

这让 RepoMind 从“分析一次、问一次”的 Demo，变成了可以长期使用的项目理解工作台。

十九、Agent trace 是什么

Agent trace 是后端关键节点日志，用来观察 Agent 执行过程。

比如：

[EntryRouter] route: agent
[Agent] question: ...
[Agent] tool_calls: [...]
[Collect] sources: [...]
[Grade] is_relevant: True
[Grade] reason: ...
[Answer] answer length: ...

项目导读问题则会看到：

[EntryRouter] route: report_answer
[ReportAnswer] use project_report directly

这些日志可以帮助我判断：

1. 是否触发 Tool Calling。
2. 调用了哪些工具。
3. 检索到了哪些来源文件。
4. 检索结果是否相关。
5. 是否触发 query rewrite。
6. 最终回答是否生成。

二十、Eval 做了什么

我设计了最小 Eval 测试集和回归脚本。

主要验证：

1. 是否成功返回 answer。
2. answer 是否非空。
3. 是否包含 expected keywords。
4. 是否包含参考文件。
5. 是否出现异常。

测试场景覆盖：

1. 项目概览。
2. 架构流程。
3. API 路由。
4. 数据库逻辑。
5. 配置定位。
6. 模糊问题。

它目前还不是复杂评测系统，但可以用于回归验证核心能力是否稳定。

二十一、项目目前的不足

当前项目还有一些限制：

1. 主要检索方式还是 Chroma 语义检索。
2. Source citation 目前是文件级，还没有做到行号级。
3. Query rewrite 当前最多重试一次。
4. Eval 目前主要是关键词和来源检查，还不够丰富。
5. 数据库目前只保存项目和问答，还没有保存完整 tool_calls、sources、grade、rewrite
