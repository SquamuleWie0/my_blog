# 环境确认与架构理解

## 一、先确认服务器环境

在开始部署和迁移之前，先确认服务器上的基础环境是否可用，尤其是 `Node.js`、`npm`、`Docker` 这些关键依赖。  
因为 VitePress 本身是基于 Node.js 的构建工具，如果环境不完整，后面无论是本地构建还是服务器部署，都会出问题。

### 1. 服务器系统特点

这台服务器是 **OpenCloudOS**，表面上看起来和 CentOS 很像，但实际上在安装方式上并不完全一致。  
我一开始尝试直接使用 NodeSource 的安装脚本，但发现它并不完全支持 OpenCloudOS，所以最后没有继续走那条路线，而是改用更通用、更稳妥的方式安装 Node.js。

### 2. 通用方式安装 Node.js

我采用的是 **官方预编译包手动安装** 的方式，这样不依赖系统仓库，也不依赖特殊安装脚本，兼容性更高。

#### 第一步：下载 Node.js 官方 Linux x64 包

```bash
cd /usr/local/src  
curl -O https://nodejs.org/dist/v20.7.0/node-v20.7.0-linux-x64.tar.xz
```

### 这一步在干什么

- `cd /usr/local/src`：进入一个适合放安装包的目录。
- `curl -O ...`：从 Node.js 官方网站下载 Linux x64 的预编译压缩包。

这个方式最直接，不依赖系统版本，也不会受发行版脚本兼容性影响。

---
#### 第二步：解压到指定目录

```bash
mkdir -p /usr/local/node  
tar -xJf node-v20.7.0-linux-x64.tar.xz -C /usr/local/node --strip-components=1
```

- `mkdir -p /usr/local/node`：创建 Node.js 的安装目录。
- `tar -xJf ...`：解压 `.tar.xz` 文件。
- `--strip-components=1`：去掉压缩包最外层目录，只保留里面真正的程序文件。

这样可以把 Node.js 放到一个固定路径里，后续维护和配置 PATH 都更方便。

---

#### 第三步：把 Node.js 加入系统 PATH
```bash
echo 'export PATH=/usr/local/node/bin:$PATH' >> /etc/profile  
source /etc/profile
```

- 第一行：把 Node.js 的执行路径写入系统环境变量。
- 第二行：立即加载配置，不用重启服务器。

如果不加到 PATH 里，系统就找不到 `node` 和 `npm` 命令。

---

#### 第四步：验证是否安装成功
```bash
node -v  
npm -v
```

检查 Node.js 和 npm 是否已经可以正常执行。

只要能正确输出版本号，就说明安装成功。

---

### 二、Docker Compose / Docker 拉镜像问题

在环境配置过程中，我还遇到了 `docker pull nginx` 失败的问题。  
一开始我误以为是 CPU 架构不兼容，后来查资料后发现，这个判断并不准确。更常见的原因其实是：
- Docker 没有装好
- 网络访问 Docker Hub 太慢
- 镜像源没有加速
- 国内访问 Docker 官方源不稳定

如果 `docker pull` 很慢或者失败，应该优先考虑
1. 检查 Docker 是否正常运行
2. 配置镜像加速器
3. 再重新拉取镜像

这类问题通常不是“镜像不能用”，而是**网络和源的问题**。  
所以解决思路也应该是先改善网络访问条件，而不是先怀疑架构本身。

### 三、同步上传博客时遇到的 SSH 问题

在把博客同步到服务器时，我还写过一个自动化脚本，但在上传这一步卡住了。  
问题主要集中在 SSH 配置上：
- 电脑没有生成 SSH key
- 密钥粘贴不完整
- 私钥没有被正确识别
- 配置文件里可能有两条指令被注释掉了

自动化脚本本身没有坏，真正卡住的是 **SSH 认证链路没有打通**。  
也就是说，脚本依赖“免密登录”或“密钥认证”，但这个前提没有配置完成。

应该按顺序检查
1. 是否生成了 SSH key
2. 公钥是否正确配置到了目标平台 
3. 私钥是否保存完整
4. `~/.ssh/config` 是否配置正确
5. 是否能手动 SSH 登录服务器

**总结**
自动化不是“写个脚本就结束”，而是要先把脚本依赖的环境条件全部准备好很关键。

## 四、VitePress 的结构理解

>我真正需要的不是博客系统，而是技术文档系统

这一段是整个迁移过程中最重要的认知变化。
### 1. 最开始的问题

我最初以为自己是在做一个博客，所以一开始一直在找 Hexo 的主题、模板和博客风格。  
但做着做着我发现，问题根本不在主题，而在于**我的需求本身就不是传统博客**。

我真正想要的是：
- 文件树结构清晰
- 分类明确
- 内容易于归档
- 每个技术方向都能独立管理
- 更像一个技术文档集，而不是一个发布型博客
### 2. 为什么 VitePress 更适合我

VitePress 更适合这种需求，因为它本质上就是：

> Markdown 内容 + 结构化 sidebar + 首页展示

它不是强迫你使用“博客模板”，而是让你自己定义内容结构。

### 3. 你现在这套结构可以这样理解
```
vitepress/  
├── docs/  
│   ├── index.md  
│   ├── guide/  
│   │   ├── 工程工具/  
│   │   ├── 基础知识/  
│   │   ├── 技术学习/  
│   │   ├── 课程笔记/  
│   │   └── 项目实践/  
│   └── .vitepress/  
│       ├── config.js  
│       └── sidebar.js
```
 
`index.md`:首页，只负责展示入口和整体介绍。

`guide/`:正文内容区，所有文章都放这里。

`.vitepress/config.js`:全局配置文件，控制网站标题、导航栏、侧边栏等。

`.vitepress/sidebar.js`: 左侧目录结构配置文件，控制文章怎么分组、怎么展示、能不能折叠。

---

## 五、三者之间的关系

### 1. `docs/guide/` 是内容

这里放的是文章本身，也就是 Markdown 文件。

### 2. `sidebar.js` 是结构

它决定文章怎么分组、怎么展开、怎么显示在左侧导航里。

### 3. `index.md` 是入口

它负责首页展示，不负责完整目录。

### 这三者不要混在一起理解

你之前之所以会觉得乱，很大一部分原因就是把：

- 文件夹结构
    
- sidebar 结构
    
- 首页结构
    

混成了一件事。  
实际上它们是三个不同层次的内容。

## 总结

在环境配置和架构梳理过程中，我逐渐意识到，这个项目的目标并不是搭建一个传统博客，而是构建一个结构清晰、便于归档和展示的技术文档系统。  
因此，我最终选择了更适合 Markdown 文档组织方式的 VitePress，并开始围绕“内容层、结构层、展示层”重新设计整个博客体系。  
同时，在服务器环境配置中，我通过通用方式手动安装了 Node.js，解决了 OpenCloudOS 上安装脚本兼容性不足的问题，也逐步理清了 Docker、SSH 和 npm 镜像加速等环境问题。
